ExamSentinel

A Secure Online Examination System with OS-Level Lockdown and VM Detection

Problem Statement

Online examinations have become widespread, but existing platforms lack robust mechanisms to prevent cheating at the operating system level. Students can easily bypass browser-based proctoring by:

  • Alt-tabbing to search for answers in other applications
  • Using screen-sharing or remote desktop tools to get external help
  • Running the exam inside a Virtual Machine to bypass all monitoring
  • Using clipboard to copy-paste from external sources
  • Opening multiple monitors to view reference material
  • Using screen recording software to capture and share questions

Browser-based solutions are fundamentally limited because they cannot control what happens outside the browser window. A determined student can always minimize, switch applications, or run the browser inside a VM where all monitoring is rendered useless.

Our Solution

ExamSentinel is a native Windows desktop application that enforces exam integrity at the operating system level. Unlike browser-based proctoring tools, ExamSentinel operates with administrator privileges and directly controls the Windows environment during an exam session.

The system consists of:

  • Desktop Client (Windows .exe) — Native Tkinter application with OS-level lockdown capabilities, built with Python and distributed as a standalone executable.
  • Backend Server (Flask REST API) — Handles authentication, exam management, session tracking, incident logging, and real-time monitoring. Deployed on Railway with PostgreSQL.
  • Teacher Dashboard — Allows teachers to create courses, author MCQ exams, activate exams, view student sessions, and review incident logs in real-time.
  • Student Interface — Pre-exam integrity checks (VM detection), exam taking under lockdown, and automatic incident reporting.

OS Lockdown System

When a student begins an exam, ExamSentinel activates a multi-layered lockdown system that restricts the operating system environment. All violations are logged with timestamps and sent to the server for teacher review.

Fullscreen Enforcement

Forces the exam window to remain in fullscreen mode. Any attempt to minimize, resize, or move the window is immediately detected and reversed. Logged as FULLSCREEN_BREACH.

Focus Monitoring

Continuously monitors which window has focus. If the student clicks or switches to any other application, the event is logged with the target window's title. Logged as FOCUS_LOST.

Keyboard Blocking

Installs a low-level keyboard hook that intercepts and blocks dangerous key combinations: Win key, Alt+Tab, Alt+F4, Ctrl+Alt+Del, PrintScreen, and all snipping tool shortcuts. Logged as KEYBOARD_BLOCKED.

Mouse Boundary

Tracks cursor position and detects if the mouse moves outside the exam window boundaries (e.g., to a taskbar or second screen edge). Logged as MOUSE_ESCAPE.

Clipboard Scrubbing

Periodically monitors the system clipboard. If any content is detected (text, images, files), it is immediately wiped and the event is logged with the format type. Logged as CLIPBOARD_SCRUB.

Process Killing

Scans running processes against a blacklist of screen recorders, remote desktop tools, and cheating software (OBS, TeamViewer, AnyDesk, Discord screen share, etc.). Detected processes are force-terminated. Logged as BLACKLISTED_PROCESS_KILLED.

Right-Click Suppression

Blocks right-click context menus within the exam window to prevent copy-paste and inspect-element style actions. Logged as RIGHT_CLICK_BLOCKED.

Multi-Monitor Detection

Detects if multiple monitors are connected. If a second display is found during the exam, the session is immediately aborted. Logged as MULTI_MONITOR_DETECTED (critical).

Standard VM Detection

Before the exam begins, ExamSentinel runs a pre-exam integrity check that scans for obvious virtualization indicators. This catches students running default or lightly configured virtual machines (VirtualBox, VMware, Hyper-V, QEMU, Parallels).

1. Process Scanning

Checks running processes against known VM guest tool process names:

  • VirtualBox: VBoxService.exe, VBoxTray.exe, VBoxClient.exe
  • VMware: vmtoolsd.exe, vmwaretray.exe, vmwareuser.exe
  • Hyper-V: vmcompute.exe, vmms.exe
  • QEMU: qemu-ga.exe

2. Registry Key Detection

Scans Windows registry for VM-specific keys:

  • SOFTWARE\Oracle\VirtualBox Guest Additions
  • SOFTWARE\VMware, Inc.\VMware Tools
  • SYSTEM\ControlSet001\Services\VBoxGuest
  • SYSTEM\ControlSet001\Services\VBoxMouse
  • SYSTEM\ControlSet001\Services\VBoxVideo

3. WMI Hardware Strings

Queries Win32_ComputerSystem and Win32_BIOS via WMI for manufacturer/model strings containing "virtualbox", "vmware", "innotek", "qemu", etc.

4. MAC Address OUI

Checks network adapter MAC address prefixes (OUI) against known VM vendor ranges:

  • 08:00:27, 0A:00:27 — VirtualBox
  • 00:0C:29, 00:50:56 — VMware
  • 00:15:5D — Hyper-V
  • 52:54:00 — QEMU

Limitation: All four methods above can be bypassed by a technically skilled student who removes guest additions, spoofs registry keys, changes MAC addresses, and modifies WMI strings. This is where Stealth VM Detection becomes essential.

Stealth VM Detection

Even when a student removes all guest tools, spoofs BIOS strings, and changes MAC addresses, ExamSentinel can still detect the virtual machine using hardware-level signals that cannot be faked from user-mode. These three methods exploit fundamental properties of how CPU virtualization works.

1. RDTSC Timing (CPU Trap Latency)

This is the most reliable detection method. It measures the CPU cycle cost of executing a CPUID instruction using the RDTSC (Read Time-Stamp Counter) instruction.

On bare metal, CPUID executes in 50-150 CPU cycles. On a hypervisor, the CPUID instruction causes a mandatory VM-exit trap to the hypervisor, which adds 3,000-50,000+ cycles of overhead. This is a fundamental architectural property of x86 virtualization that no VBoxManage command or registry tweak can eliminate.

ExamSentinel injects x86-64 shellcode into executable memory, runs 200 iterations, takes the median, and compares against a threshold of 2,500 cycles.

2. Thermal Zone Absence

Every real PC has at least one thermal sensor exposed through the WMI MSAcpi_ThermalZoneTemperature class. These sensors report CPU/motherboard temperature readings that fluctuate naturally with load.

Virtual machines have no physical hardware, so they either report zero thermal zones or return a constant stub value (exactly 26.85C / 300K). ExamSentinel queries this WMI namespace and flags the VM if no zones exist or if readings are suspiciously constant.

3. SCSI Disk Identifier

Virtual disk controllers report identifiers like "VBOX HARDDISK" or "VMware Virtual disk" in the Windows registry under HARDWARE\DEVICEMAP\Scsi. These identifiers come from the virtual storage controller firmware, not from installed tools.

Removing Guest Additions, spoofing BIOS strings, or changing MAC addresses does NOT affect the SCSI disk identifier. It can only be changed by modifying the hypervisor source code itself.

System Architecture

Technology Stack

  • Client: Python, Tkinter, ctypes, psutil, pyinstaller (Windows .exe)
  • Server: Flask, SQLAlchemy, PostgreSQL, Flask-JWT-Extended
  • Deployment: Railway (server + database), standalone .exe (client)

Exam Flow

  1. Teacher creates course, adds students, authors exam with MCQs
  2. Teacher activates exam when ready
  3. Student launches ExamSentinel, logs in, sees available exams
  4. Student starts exam — pre-exam integrity check runs (VM detection)
  5. If passed, session transitions to in_progress, lockdown engages
  6. Student answers questions under full OS lockdown
  7. All violations logged in real-time to server
  8. Student submits — lockdown disengages, results shown
  9. Teacher reviews sessions, scores, and incident timeline
